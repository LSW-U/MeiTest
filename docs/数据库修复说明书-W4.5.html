二、P0-1：订单列表 events 优化

问题

列表场景（listUserOrders + listAllOrders）include.events 不带 take，每页 20 单 × 10 events = 200 行全量拉回。

// 修复前
include: {
  items: { orderBy: { id: 'asc' } },
  events: { orderBy: { createdAt: 'asc' } },  // ← 全量
}
修复
// 列表场景（只拉最新一条）
events: { orderBy: { createdAt: 'desc' }, take: 1 }

// 详情场景（保留完整 timeline）
events: { orderBy: { createdAt: 'asc' } }

文件：apps/api/src/modules/order/order.service.ts（4 处修改）

三、P0-2：rider_locations 死表

问题
  • WS location:update handler 只 emit 不写库
  • Dashboard countOnlineRiders 查 rider_locations 表 → 永远 0
  • 两张死表浪费空间
修复（方案 A：查 Redis）

W3-C 已有 Redis 在线状态：rider:online:{riderId} SETEX 60s

// 修复后
private async countOnlineRiders(now: Date): Promise {
  const keys = await redis.keys('rider:online:*');
  return keys.length;
}

删除：schema.prisma 的 RiderLocation + RiderLocationHistory model + RiderProfile 关系字段

Migration：20260628000000_drop_rider_locations_w

DROP TABLE IF EXISTS "rider_location_histories" CASCADE;
DROP TABLE IF EXISTS "rider_locations" CASCADE;

四、P0-3：复合索引

问题

admin /orders?status=PENDING_CONFIRM 走单列 orders_status_idx，ORDER BY createdAt 需额外 sort。

修复
-- migration 20260628010000_add_orders_status_created_at_idx_w
CREATE INDEX CONCURRENTLY IF NOT EXISTS "orders_status_created_at_idx"
  ON "orders" ("status", "created_at" DESC);

复合索引让 WHERE status=? ORDER BY created_at DESC 一次走索引扫描。
