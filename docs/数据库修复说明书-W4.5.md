## 二、P0-1 详细：订单列表 events 优化

### 问题

```ts
// 修复前：列表 + 详情都全量拉 events
const orders = await db.order.findMany({
  include: {
    items: { orderBy: { id: 'asc' } },
    events: { orderBy: { createdAt: 'asc' } },  // ⚠️ 没 take，全量拉
  },
});
```

每页 20 单 × 平均 10 events = **200 行 events 全量拉回**。

### 修复

**列表场景**（`listUserOrders` + `listAllOrders`）：
```ts
events: { orderBy: { createdAt: 'desc' }, take: 1 },  // 只拉最新一条
```

**详情场景**（`getOrderDetail` + `adminGetOrderDetail`）：
```ts
events: { orderBy: { createdAt: 'asc' } },  // 保留全量 timeline
```

### 文件

- `apps/api/src/modules/order/order.service.ts`：4 处 include 修改

## 三、P0-2 详细：rider_locations 死表

### 问题

- WS `location:update` handler 只 emit 不写库
- `dashboard.countOnlineRiders` 查 `rider_locations` 表 → **永远返回 0**
- `rider_locations` + `rider_location_histories` 两张表是空表，浪费空间

### 修复（方案 A：查 Redis）

**W3-C 已有 Redis 在线状态**：`rider:online:{riderId}` SETEX 60s（heartbeat 续期）

```ts
// 修复后：countOnlineRiders 改查 Redis
private async countOnlineRiders(now: Date): Promise<number> {
  const keys = await redis.keys('rider:online:*');
  return keys.length;
}
```

**删除死表**：
- schema.prisma 删 `RiderLocation` + `RiderLocationHistory` model
- schema.prisma 删 `RiderProfile.currentLocation` + `locationHistories` 关系字段
- migration `20260628000000_drop_rider_locations_w`：`DROP TABLE IF EXISTS ... CASCADE`

### 文件

- `apps/api/src/modules/platform/dashboard.service.ts`：countOnlineRiders 改 Redis
- `apps/api/prisma/schema.prisma`：删 2 个 model + 2 个关系字段
- `apps/api/prisma/migrations/20260628000000_drop_rider_locations_w/migration.sql`
- `apps/api/tests/dashboard.service.test.ts`：mock 从 riderLocation.count 改为 redis.keys

## 四、P0-3 详细：复合索引

### 问题

admin `/orders?status=PENDING_CONFIRM` 典型查询走单列索引 `orders_status_idx`，ORDER BY `createdAt DESC` 需要额外 sort。

### 修复

```sql
-- migration 20260628010000_add_orders_status_created_at_idx_w
CREATE INDEX CONCURRENTLY IF NOT EXISTS "orders_status_created_at_idx"
  ON "orders" ("status", "created_at" DESC);
```

复合索引让 `WHERE status=? ORDER BY created_at DESC` 一次走索引扫描。
