# MeiMart 三流程并行开发方案

> 把 21 个后端代码模块切成三条可并行流程,每条说清楚做完能跑通什么。
> 共享前置层就绪后,三条业务流程可立即并行启动,互不阻塞。

---

## 一、后端所有代码模块清单(共 21 个)

### A. 基建/共享层(5 个,前置必做)

| #   | 模块             | 职责                        |
| --- | -------------- | ------------------------- |
| 0   | `api-contract` | OpenAPI 契约源 + 类型生成        |
| 0   | `shared-db`    | PG 连接、迁移、事务封装             |
| 0   | `shared-cache` | Redis 封装(缓存/会话/Pub-Sub)   |
| 0   | `shared-infra` | OSS / 短信 / 地图 / 推送 SDK 封装 |
| 0   | `gateway`      | 鉴权中间件、限流、错误处理、日志、trace    |

### B. 业务模块(17 个)

| 域 | 模块 |
|---|---|
| **身份域** | `auth` · `user` |
| **供给域** | `shop` · `catalog` · `inventory` · `pricing` |
| **交易域** | `cart` · `order` · `payment` · `refund` |
| **配送域** | `dispatch` · `rider` · `location` |
| **沟通域** | `im` · `notify` |
| **治理域** | `settle` · `platform` |

---

## 二、共享前置(三条线启动前 W1 必须就绪)

```
契约 → monorepo 脚手架 → 基建 5 模块 → 数据库基线 schema → 鉴权框架
```

**就绪标准**:三端能 mock 出登录,任意模块能 `import { db, cache, oss } from '@shared'`,契约类型可生成。

完成此层后,**三条业务流程可立即并行启动**,互不阻塞。

## 三、三条并行流程

### 流程 1 — 供给、仓储与浏览线 (商家+仓库 → 用户浏览)

**覆盖模块**:`auth` · `user` · `shop`(**单一商家**) · **`warehouse`(新增)** · `catalog` · `inventory`(**按仓库**) · `pricing`(**按仓库配送费**)

**对外契约**:身份、地址、店铺、**仓库**、商品、库存(商品×仓库)、价格相关接口

**完成后能实现什么**:
- 三端用户登录注册(RBAC + **视角切换器**)
- 用户能维护收货地址
- **单一商家**店铺信息维护(预置1条,入驻接口留口不开放)
- **多仓库 CRUD**(5~10 个,各含地理坐标、覆盖范围、营业时间、配送费)
- **按仓库的库存管理**(商品×仓库维度,行锁防超卖)
- 商品 CRUD、SKU/分类、上下架
- **按仓库的配送费规则配置**
- 客户端 App 浏览商品,**切换收货地址时库存实时变化**(PostGIS 自动匹配最近仓库)
- 平台能配置全局费率/抽成/起送规则
- super_admin 通过**视角切换器**在 5 视角间切换(平台/商家/仓库/客服/骑手管理)

**总结**:从"商家上架商品 + 多仓库备货"到"用户能浏览选购 + 下单自动匹配仓库"的完整链路打通。

**依赖**:仅依赖前置层。无跨流程阻塞。

---

### 流程 2 — 交易与配送线 (下单 → 支付 → 配送 → 送达)

**覆盖模块**:`cart` · `order` · `payment` · `refund` · `dispatch` · `rider` · `location` · `notify`

**对外契约**:购物车、订单、支付、退款、配送、骑手、位置、通知相关接口

**完成后能实现什么**:
- 客户端购物车增删改、跨店铺分组、结算前价格库存校验
- 客户端下单走完整状态机(`PENDING_CONFIRM → ... → COMPLETED`),同步事务防超卖
- 微信支付下单与回调幂等处理,订单超时自动取消
- 商家在后台 web 接单/拒单/备餐完成
- 骑手端 App 上线打卡、抢单/接派单、取餐送达、异常上报
- 骑手位置实时上报,客户端能在地图上看骑手位置和 ETA
- 关键节点(下单/接单/取餐/送达)触发 Push + 短信 + 站内信
- 商家接单前可全额退款、接单后由商家决定退款

**一句话**:一笔订单从"用户点结算"到"骑手送达"的核心交易闭环跑通。

**依赖**:
- 商品/店铺/价格数据 → 从流程 1 拿(前期可 mock)
- RBAC 与骑手身份 → 从前置层 + 流程 1 的 `auth` 拿
- 实际联调在流程 1 完成后即可,但**开发期可独立 mock**

---

### 流程 3 — 治理与财务线 (平台管理 + 结算 + 沟通)

**覆盖模块**:`platform` · `settle` · `im` · `audit`(审计并入 platform)

**对外契约**:平台运营、结算、报表、IM、审计相关接口

**完成后能实现什么**:
- 后台 web 平台角色看数据看板(GMV / 在线商家 / 在线骑手 / 异常订单)
- 平台能管理商家与骑手(审核、封禁、解封)
- 平台能配置全局系统参数(开关、费率、风控规则)
- 所有敏感操作有审计日志可追溯
- 客户 ↔ 商家 / 客户 ↔ 骑手 / 客户 ↔ 客服 三类会话能跑(三方 IM 托管 + 后端薄壳)
- 商家日终结算单自动生成,骑手佣金按订单计算
- 商家/骑手能发起提现申请,平台能审核打款(线下)
- T+1 手动结算走通,流水清晰可对账

**一句话**:平台能"看得见、管得住、算得清",商家和骑手能"拿到钱"。

**依赖**:
- 订单/配送数据 → 从流程 2 拿(前期可 mock 历史订单)
- 商家/骑手数据 → 从流程 1 + 前置层拿
- IM 用户身份签名 → 从 `auth` 拿

```
                                                        │

                                                        │

```

## 六、流程间联调要点

| 联调项 | 涉及流程 | 时点 | 关键校验 |
|---|---|---|---|
| 商品 → 购物车 → 下单 | 1 ↔ 2 | W5 初 | 价格、库存、上下架一致性 |
| **收货地址 → 仓库匹配 → 库存** | 1 ↔ 2 | W5 初 | PostGIS 匹配正确,跨仓库不串库存 |
| 商家接单 → 骑手抢单 | 1(仓)+ 2(单) | W5 中 | warehouse_id 路由正确,骑手按仓分组 |
| 订单 → 结算单 | 2 → 3 | W6 初 | 佣金、抽成、退款冲减 |
| IM 会话身份 | 1(auth)+ 3(im) | W6 中 | 三方 IM 签名与 RBAC 一致 |
| 视角切换 → 数据范围 | 1 + 3 | W6 中 | 各视角看到的数据范围正确,审计可追 |
| 审计覆盖 | 全流程 → 3 | W7 | 关键操作都有日志 |
