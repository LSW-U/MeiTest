## 一、项目基本目的与背景

### 1.1 项目概述

Mei Mart 是面向东帝汶市场的本地电商平台，首期聚焦超市零售场景，为消费者提供手机端购物体验。

### 1.2 项目目的

| 目标 | 说明 |
|------|------|
| 顾客端 | 商品浏览、搜索、加入购物车、下单、支付、订单追踪 |
| 商家端（后期） | 商品管理、订单处理、库存管理、销售统计 |
| 本地化服务 | 适配东帝汶网络环境、支持多语言、对接本地支付方式 |

### 1.3 项目背景

- **市场机会**: 东帝汶零售业数字化程度低，移动互联网渗透率逐步上升
- **网络现状**: 首都帝力4G覆盖较好，其他地区网络不稳定，需离线支持
- **语言需求**: 德顿语、葡萄牙语、印尼语、英语多语言支持
- **支付习惯**: 以现金支付为主，电子支付（M-Pesa等）逐步普及
- **文化特征**: 融入东帝汶传统纹样（Tais）、传统建筑（Uma Lulik）等文化元素

### 1.4 当前阶段

原型阶段：已完成 26 个前端页面的 HTML 原型 + Vue 3 SPA 框架搭建，正在进行技术架构升级和AI辅助全栈开发。

### 2.2 前端技术栈

| 类别 | 选型 | 版本 | 选型理由 |
|------|------|------|----------|
| 开发框架 | React Native + Expo | SDK 52+ | AI训练数据最丰富，跨平台一套代码 |
| 语言 | TypeScript | 5.x | 类型安全，AI生成代码准确率高 |
| UI样式 | NativeWind | 4.x | Tailwind CSS风格，AI理解准确 |
| 组件库 | 自建组件库 | - | 基于原型设计系统定制 |
| 状态管理 | Zustand | 5.x | 极简API，AI生成代码质量高 |
| 网络请求 | tRPC + React Query | 11.x / 5.x | 端到端类型安全 |
| 离线存储 | WatermelonDB | 0.27+ | React Native原生离线数据库 |
| 路由导航 | Expo Router | 4.x | 文件系统路由，约定式配置 |
| 多语言 | i18next | 24.x | AI训练数据最多的国际化方案 |
| 图片处理 | expo-image | 3.x | 渐进式加载、缓存 |
| 图标 | Material Symbols | - | 与原型保持一致 |

| 类别 | 选型 | 版本 | 选型理由 |

| API风格 | tRPC + RESTful | 11.x | tRPC内部使用，RESTful对外 |

## 四、代码结构说明

### 4.1 前端架构分层

```
┌──────────────────────────────┐
│         Pages (页面层)        │  ← Expo Router 文件路由
│       app/**/*.tsx            │
├──────────────────────────────┤
│     Components (组件层)       │  ← 可复用UI组件
│   src/components/**           │
├──────────────────────────────┤
│      Hooks (逻辑层)           │  ← 可复用业务逻辑
│   src/hooks/**                │
├──────────────────────────────┤
│     Store (状态层)            │  ← Zustand 全局状态
│   src/store/**                │
├──────────────────────────────┤
│    Services (服务层)          │  ← API调用封装
│   src/services/**             │
├──────────────────────────────┤
│    Database (离线层)          │  ← 本地数据库
│   src/database/**             │
└──────────────────────────────┘
```

### 4.2 数据流规范

**正常场景**:
```
用户操作 → Page 组件
         → Hook (封装业务逻辑)
         → Service (发起 tRPC 请求)
         → 后端处理
         → 响应更新 Store
         → Store 触发组件重新渲染
```

**离线场景**:
```
用户操作 → Page 组件
         → Hook
         → Database (本地写入)
         → 同步队列
         → 网络恢复后 → Service → 后端
```

### 4.3 组件分类原则

| 类型 | 位置 | 特点 | 示例 |
|------|------|------|------|
| 基础组件 | components/ui/ | 无业务逻辑，高度可复用 | Button, Input, Modal |
| 业务组件 | components/business/ | 包含特定业务逻辑 | ProductCard, OrderCard |
| 布局组件 | components/layout/ | 页面结构 | BottomNav, PageHeader |
| 反馈组件 | components/feedback/ | 状态提示 | EmptyState, OfflineBanner |

### 4.4 后端架构分层

```
┌──────────────────────────────┐
│       Routers (路由层)        │  ← tRPC 路由定义
│    src/routers/**             │
├──────────────────────────────┤
│     Procedures (过程层)       │  ← 输入校验 + 调用Service
│    src/procedures/**          │
├──────────────────────────────┤
│     Services (服务层)         │  ← 核心业务逻辑
│    src/services/**            │
├──────────────────────────────┤
│    Prisma (数据访问层)        │  ← 数据库操作
│    prisma/schema.prisma       │
└──────────────────────────────┘
```

## 五、组件库设计

### 5.1 设计原则

- **复用优先**: 通用逻辑抽取为组件
- **单一职责**: 一个组件只做一件事
- **Props驱动**: 行为通过Props配置，避免内部硬编码
- **类型安全**: 所有Props必须有TypeScript接口定义
- **AI友好**: 每个组件有JSDoc注释，方便AI理解和使用

#### 6.1.1 基础配置

| Authentication | Bearer Token (JWT) |
| Char Encoding | UTF-8 |

#### 6.1.2 通用响应结构

**成功响应**:
```json

  "success": true,
  "code": 200,
  "message": "操作成功",
  "data": {},
  "meta": {
    "page": 1,
    "pageSize": 20,
    "total": 100,
    "totalPages": 5
  }
}
```

**错误响应**:
```json
{
  "success": false,
  "code": 400,
  "message": "参数校验失败",
  "errors": [
    { "field": "email", "message": "邮箱格式不正确" }
  ]
}
```

#### 6.1.3 状态码规范

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 200 | 成功 | 正常请求 |
| 201 | 已创建 | 资源创建成功 |
| 400 | 请求错误 | 参数校验失败 |
| 401 | 未认证 | Token过期或无效 |
| 403 | 无权限 | 权限不足 |
| 404 | 未找到 | 资源不存在 |
| 409 | 冲突 | 重复操作（如重复注册） |
| 429 | 请求过多 | 触发限流 |
| 500 | 服务器错误 | 内部异常 |
