# MeiMart 后端代码模块与三端功能拆分

> 模块拆分三个视角:**后端业务域**(真正解耦的模块)、**三端功能模块**(前端拆页)、**共享基建**(monorepo 复用层)。

---

## 一、后端业务模块(共 ~21 个)

| # | 模块 | 职责 | MVP | 关键表 |
|---|---|---|---|---|
| 1 | `auth` | 三端登录、JWT、RBAC、device_type | ✅ | users, roles, user_roles, sessions |
| 2 | `user` | 用户资料、收货地址、头像 | ✅ | user_profiles, addresses |
| 3 | `catalog` | 商品、SKU、分类、规格、上下架 | ✅ | products, skus, categories, specs |
| 4 | `shop` | **单一商家**(预置1条记录)、入驻接口留口 | ✅ | shops(1 条记录) |
| 5 | `warehouse` | **多仓库**(5~10个,地理位置)、覆盖范围、营业时间、配送费 | ✅ | warehouses, delivery_zones |
| 6 | `inventory` | **商品×仓库** 维度库存、行锁防超卖、变更日志 | ✅ | stocks(product_id+warehouse_id), stock_logs |
| 7 | `cart` | 购物车(可 Redis)、合并结算 | ✅ | carts, cart_items |
| 8 | `order` | 订单状态机(带 warehouse_id)、取消、备注、评价 | ✅ | orders, order_items, order_events |
| 9 | `payment` | 微信/支付宝、回调、对账 | ✅ | payments, payment_logs |
| 10 | `refund` | 退款申请、审核、回款 | ⚠️ | refunds |
| 11 | `pricing` | 价格、活动价、平台抽成规则、**按仓库的配送费** | ✅ | price_rules, commissions |
| 12 | `dispatch` | 抢单/派单(**按仓库分组**)、超时重派、配送费 | ✅ | deliveries, dispatch_tasks |
| 13 | `rider` | 骑手资料、在线状态、班次、保证金 | ✅ | riders, rider_status, shifts |
| 14 | `location` | 位置上报、轨迹存储、地理围栏 | ✅ | driver_locations, tracks |
| 15 | `im` | 会话、消息、未读(三方托管则薄壳) | ✅ | conversations, messages |
| 16 | `notify` | Push、短信、站内信、模板管理 | ✅ | notifications, notify_templates |
| 17 | `settle` | 商家结算、骑手佣金、平台抽成、提现 | ⚠️ | settlements, withdraws |
| 18 | `marketing` | 优惠券、满减、新人券、首页 banner | 🔻 后置 | coupons, promotions |
| 19 | `platform` | 平台运营:审核、风控、配置、审计、**视角切换** | ✅ | audit_logs, sys_configs |
| 20 | `search` | 商品/订单搜索(MVP 用 PG ILIKE) | 🔻 后置 | — |
| 21 | `analytics` | 报表、漏斗、BI | 🔻 后置 | — |

**图例**: ✅ 必做  ⚠️ 简化版  🔻 后置

### 模块间通信规则

- 同进程**只能**通过 service 调用或事件总线,严禁跨模块直连表
- 跨模块事件示例:`OrderPaid → dispatch.派单 + notify.通知商家`
- 事件总线 MVP 用 Node EventEmitter + Redis Pub/Sub 双写(为将来拆服务留口)

## 二、三端功能模块(前端视角)

### 客户端 App(8 个功能页)

```
auth     登录/注册/微信
home     首页(分类/轮播/推荐)
shop     店铺详情/商品列表/搜索
cart     购物车/结算
order    下单/订单列表/订单详情/跟踪
im       客服/骑手会话
profile  个人中心/地址/钱包/优惠券
after    退款/售后/评价
```

### 骑手端 App(6 个功能页)

```
auth     登录/实名/保证金
duty     上下班/班次/接单模式(抢单 or 派单)
task     待抢/进行中/已完成 订单列表
deliver  取餐/送达/拍照/异常上报
earning  收入明细/提现
im       客户/商家/客服会话
```

### 后台 Web(super_admin 通过视角切换看不同维度)

顶部 [视角切换器: 平台 ▼]
```
                ├─ 平台(默认)
                ├─ 商家
                ├─ 仓库
                ├─ 客服
                └─ 骑手管理
```

各视角菜单:
```
平台视角:
  dashboard  数据看板(GMV/订单/在线骑手/异常订单)
  rider      骑手审核/列表/封禁
  settle     结算单/提现审核
  config     配置(抽成/营业规则/系统开关)
  audit      审计日志/风控

商家视角(单一商家):
  home       店铺概览
  product    商品/分类/库存
  order      待接单/进行中/已完成/退款
  shop       店铺信息/公告
  finance    收入/对账
  im         客户会话

仓库视角:
  list       仓库列表(5~10 个)
  stock      按仓库的库存管理
  zone       配送范围/营业时间
  fee        各仓库配送费规则

客服视角:
  ticket     客户工单
  order      订单查询(全局)
  user       用户资料查询

骑手管理视角:
  list       骑手列表
  earning    收入明细
  track      轨迹回放
  audit      审核/封禁
```

**关键澄清**:视角切换 ≠ 切账号,同一 super_admin token。切换只改前端路由 + 数据过滤,后端 RBAC 不感知。**不做** impersonate。

## 三、共享基建(monorepo 复用层)

```
packages/
├── api-contract/        ← OpenAPI 源文件,所有端的契约源头
├── shared-types/        ← 从契约生成的 TS 类型
├── shared-utils/        ← 时间/金额/校验工具(三端共用)
├── shared-validators/   ← zod schema(前后端共用校验)
└── ui-kit/              ← 后台 web 用的组件库(App 用各自原生)

apps/
├── api/                 ← 后端(包含上面 17 个业务模块)
├── client-app/          ← 客户端 App(React Native/uniapp)
├── rider-app/           ← 骑手端 App
└── admin-web/           ← 后台 Web(Next.js)
```

后端 `apps/api` 内部:

```
src/
├── modules/
│   ├── auth/
│   │   ├── auth.controller.ts
│   │   ├── auth.service.ts
│   │   ├── auth.repository.ts
│   │   ├── auth.dto.ts
│   │   ├── auth.events.ts
│   │   └── auth.module.ts
│   ├── catalog/
│   ├── order/
│   └── ...
├── shared/              ← 跨模块共享(db/client/middleware/decorator)
├── infrastructure/      ← redis/oss/sms/map sdk 封装
└── main.ts
```

## 四、推荐推进顺序(对应模块)

```
Phase 1 基建(W1)
  api-contract + monorepo 脚手架 + CI + Docker

Phase 2 用户域(W2)
  auth + user + shop(基础信息) + **视角切换器**  ← RBAC + 视角切换在此落地

Phase 3 商品与仓储域(W3)
  catalog + **warehouse** + inventory(**按仓库**) + pricing(**按仓库配送费**)

Phase 4 交易域(W4-W5)
  cart + order + payment  ← 状态机、防超卖(warehouse_id)、回调幂等

Phase 5 配送域(W6)
  dispatch + rider + location + WebSocket(**按仓库分组派单**)

Phase 6 沟通域(W7)
  im + notify + 后台 web 多视角菜单完善

Phase 7 运营域(W8)
  settle + platform + refund + 上线
```

## 五、模块数量结论

- **后端业务模块**: MVP 必做 **13 个**(含新增 `warehouse`), 简化 2 个, 后置 3 个, 共 **~18 个**
- **前端功能页**: 客户端 **8** + 骑手 **6** + 后台 Web(**5 视角 30+ 菜单项**) = **40+ 页**
- **monorepo 包**: 5 个共享 + 4 个应用 = **9 个包**
