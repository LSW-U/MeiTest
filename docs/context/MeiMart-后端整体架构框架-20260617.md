# MeiMart 后端整体架构框架

> 全栈架构视角下的 MeiMart 后端 MVP 整体框架,从分层 → 模块 → 数据流 → 推进顺序四块展开。
> 三端形态:客户端 App + 骑手端 App + 后台 Web(平台运营+商家多角色)。
> **本文档已同步契约 v0.3 决策(2026-06-19),冲突处以 `API契约文档-v0.3.md` 为准。**

---

## 一、分层架构

```
┌─────────────────────────────────────────────────┐
│  客户端 App   │  骑手端 App   │  后台 Web        │
│  (C 端用户)   │   (骑手)      │ (平台运营+商家)   │
└─────────────────────────────────────────────────┘
                      ↓ HTTPS / WSS
┌─────────────────────────────────────────────────┐
│              API Gateway (鉴权/限流/路由)         │
├─────────────────────────────────────────────────┤
│  BFF 层(可选): 按端裁剪字段,聚合接口              │
├─────────────────────────────────────────────────┤
│         业务服务层(单体或多模块单进程)             │
│  用户·商品·订单·支付·配送·IM·营销·结算           │
├─────────────────────────────────────────────────┤
│  基础设施层: 缓存·消息队列·对象存储·定位·推送      │
├─────────────────────────────────────────────────┤
│         PostgreSQL + PostGIS · Redis             │
└─────────────────────────────────────────────────┘
```

**MVP 阶段建议**:**模块化单体**(Modular Monolith)——一个仓库、一个进程、按业务模块划目录。**不要**一上来就微服务,四端还没流量,拆了只是自找麻烦。

---

## 二、模块划分(对应契约 v0.2 + v0.3 的领域)

| 模块 | 职责 | 关键表 |
|---|---|---|
| `auth` | 三端登录、JWT(`{ sub, role, deviceType, jti }`,**无 clientType**)、RBAC(5 角色)、Token 分端 TTL、logout 黑名单 | users, user_roles, audit_logs |
| `catalog` | 商品(多语言 `Record<string,string>`)、SKU、分类 | products, skus, categories |
| `shop` | **单一商家信息(预置1条)**、入驻接口留口不开放 | shops(1 条记录) |
| `warehouse` | **多仓库(5~10 个)**、PostGIS 覆盖范围多边形、营业时间、配送费 | warehouses(含 GIST 索引) |
| `inventory` | **商品×仓库 维度库存**、行锁防超卖 | stocks(warehouse_id+sku_id), stock_logs |
| `order` | 下单(**自动匹配仓库**)、orderNo 16 位(`MM+yyyyMMdd+wh2+seq4`)、状态机、取消、退款 | orders(含 warehouse_id), order_items, order_events |
| `payment` | **5 种支付方式**(COD/BANK 真实 + WECHAT mock + PAYPAL/STRIPE stub) | payment_intents, idempotency_keys |
| `dispatch` | 抢单、派单(**按仓库分组**)、**WS+轮询双轨**配送追踪 | rider_locations, rider_location_histories |
| `rider` | 骑手资料、在线状态、收入 | riders, earnings |
| `im` | 用户↔商家↔骑手会话(**MVP 阶段接口预留,延后接入**) | (预留表) |
| `notify` | SMS 验证码(stub → W6 切本地 Timor Telecom/Telkomcel)、邮件(MailHog/SendGrid)、Push(**延后**) | notifications |
| `settle` | 商家结算、骑手佣金、平台抽成 | settlements |
| `admin` | 后台审计、风控、运营配置、**视角切换**(5 视角,审计字段) | audit_logs(含 perspective + deviceType) |

每个模块**自带 controller / service / repository / dto**,模块之间通过**内部事件总线**或**显式服务调用**通信,严禁跨模块直连表。

## 三、数据流(同步 + 实时)

**同步流(HTTP)**:上下架、下单、查询、支付发起——RESTful 即可,契约驱动。

**实时流(WebSocket)**——这是外卖场景的命脉,**WS + 轮询双轨设计**(v0.3 决策 G3):
- **Socket.IO 主通道**:骑手 App 每 5s 推送位置 → 服务端 → 客户端 App 实时显示
- **HTTP 轮询兜底**:`GET /orders/:id/tracking` 间隔 30s,WS 断线时降级
- 客户端 App 启动时同时开 WS + 定时器,WS 收到消息就重置定时器
- 订单状态变更 → 客户端/后台 web 同步刷新

> 骑手按"**取货仓库**"分组派单,避免跨仓取货浪费时间。

建议:**Redis Pub/Sub + Socket.IO**,MVP 不上 Kafka/RocketMQ,等订单量上来再换。

**关键异步任务**(走 BullMQ 队列):
- 支付回调幂等
- 订单超时自动取消
- 结算日终批处理
- SMS/邮件发送重试

## 四、契约驱动的开发工作流

```
Obsidian 契约 v0.2(基础)+ v0.3(变更覆盖)
        ↓ generate
前端 types  ·  后端 dto/zod schema  ·  Mock Server
        ↓
各自实现 → 联调时契约就是测试基准
```

核心原则:**契约是 single source of truth**。改契约 v0.3 → 自动生成 zod schema → OpenAPI → 三端类型同步。MVP 阶段一人全栈,契约从"多方合同"简化为"个人备忘录 + 类型生成器"。

## 七、几个关键决策点(2026-06-19 全部更新到 v0.3)

1. **BFF 要不要**:三端字段差异大才上,MVP 阶段可以**先不要**。
2. **是否多租户**:**单一商家**架构,shops 1 条预置记录,入驻接口留口不开放。多仓库按 `warehouse_id` 隔离。
3. **状态机怎么实现**:订单状态机建议**显式 state machine 库**(如 XState 服务端版),不要 if-else 堆。
4. **幂等怎么保证**:所有外部回调(支付、推送)必须有 **IdempotencyKey 表**。
5. **仓库匹配策略**:下单时按收货地址 lat/lng 用 PostGIS 查找覆盖该点的最近仓库,匹配失败 → 422 `E-ORDER-OUT-OF-DELIVERY-RANGE`。
6. **后台 web 视角切换**:`super_admin` 同一 JWT(`{ sub, role, deviceType, jti }`,**无 clientType**),前端 zustand 持久化 `perspective`,顶部下拉切 5 视角(平台/商家/仓库/客服/骑手管理)。fetch interceptor 注入 `X-Perspective` header,**后端 RBAC 不感知**,仅 AuditLog 记录。**不做**模拟登录(impersonate)。
7. **JWT 设计**(v0.3 决策 C):**删除 clientType**,payload 只含 `{ sub, role, deviceType, jti }`。`deviceType` ∈ {`client_app`, `rider_app`, `admin_web`},由前端 App 配置写死。
8. **Token 有效期**(v0.3 决策 E):**分端 TTL** — 客户端 access 30d / 骑手 12h / 后台 2h;refresh 统一 60d。
9. **logout**(v0.3 决策 F):**必传 refreshToken**,服务端加 Redis 黑名单(`blacklist:{jti}` TTL = refresh 剩余时间),立即失效。
10. **orderNo 格式**(v0.3 决策 A):`MM + yyyyMMdd + warehouseId(2位) + 序号(4位)` = 16 位,例 `MM2026061901000234`。序号由 Redis `INCR order:seq:{date}:{whCode}` 生成,2 天过期。
11. **多语言字段**(v0.3 决策 B):统一 `Record<string, string>` JSON,例 `{"en":"Milk","zh":"牛奶","id":"Susu","pt":"Leite"}`。查询按 `Accept-Language` 取值,fallback 链 `lang → en → ""`。
12. **支付策略**(v0.3 决策 D):**全预留 5 枚举** `COD/BANK_TRANSFER/WECHAT/PAYPAL/STRIPE`,抽象 `PaymentStrategy` 接口。MVP 阶段 COD/BANK 真实流程,WECHAT/PAYPAL/STRIPE mock/stub,W6 拿主体后逐个切真。
13. **OTP 策略**(v0.3 + SMS 反转):**密码(主)+ SMS(手机验证,测试 stub)+ 邮箱(找回)**。抽象 `OtpStrategy` 接口,W6 SMS 切东帝汶本地。
14. **配送追踪**(v0.3 决策 G):**WS + 轮询双轨**,Socket.IO 主通道,HTTP 30s 兜底。
